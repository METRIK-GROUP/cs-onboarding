Como sua mudança vai para o ar

Do código ao domínio de produção em 10 passos.

Cada passo tem proteção automática. Se algo der errado em qualquer ponto, o sistema trava antes de chegar em produção — você corrige, pusha de novo, segue.

Pull da main

Sempre começa do código mais atualizado.

Branch cs/<feature>

Nome descritivo em kebab-case.

Código + pnpm dev

Faz a mudança, testa em localhost.

Lint + typecheck

pnpm lint e tsc --noEmit passam.

Commit + push

Mensagem no padrão feat/fix/refactor(cs):

Abre PR

Descreve o que mudou, por quê, como testar.

CI roda (~3 min)

Lint, build, validação de escopo, preview Vercel.

Testa no preview

URL temporária gerada pela Vercel no PR.

Avisa o Rodrigo

Quando estiver satisfeita, manda o link do PR pra ele revisar.

Rodrigo aprova + merge

Ele confere o preview, aprova e mergeia. Deploy em produção dispara sozinho.

Dados em tempo real

Sincronizando Hotmart, Asaas e TMB.

O Supabase recebe dados das APIs externas a cada 6 horas automaticamente. Quando você precisa testar com uma venda recém-feita ou uma cobrança atualizada, pode forçar o sync manualmente.

Sync automático

A cada 6h, o GitHub Actions dispara o sync das APIs que alimentam a Central CS. Você vê o histórico em github.com/METRIK-GROUP/launch-dashboard/actions.

Sync manual, quando usar

Acabou de entrar uma venda nova no Hotmart e você quer testar o fluxo de liberação de acesso imediatamente. Ou um aluno pagou uma parcela em atraso e você precisa ver o status atualizado no dashboard agora, sem esperar a próxima janela de 6h.

Como disparar manualmente

1. Actions → Sync Hotmart (ou Asaas, ou TMB)
2. Run workflow → branch main → Run workflow
3. Espera ~1 min, confere se ficou verde
4. Abre o dashboard, dados atualizados
Sync Hotmart Sync Asaas Sync TMB
Quando travar

Tudo que precisa de SOS.

Se o CI falhar com erro que você não entende, se o preview mostrar comportamento diferente do local, ou se a produção ficar estranha após seu deploy: não insista sozinha. É mais barato perguntar do que desfazer.

O rollback de emergência é um comando: npx vercel rollback. Volta para o deploy anterior em segundos.
